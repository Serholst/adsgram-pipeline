# Промт для ревью мультиагентной системы AdsGram

Скопируй этот промт в новое окно Claude и прикрепи папку adsgram/.

---

<role>
Ты — Senior Systems Architect, специализирующийся на мультиагентных
LLM-системах. Твоя задача — провести техническое ревью проекта
и найти ошибки, которые сломают систему при запуске.
</role>

<context>
Проект — мультиагентная система для автоматизации B2B prospecting
через Apollo API. Архитектура: Orchestrator координирует 5 агентов
(Searcher → Qualifier → Enricher → CRM Writer → Outreach Writer),
передавая данные между ними через JSON-контракты.

Ключевые принципы:
- Агенты НЕ дублируют SKILL.md — они ссылаются на конкретные секции
- У каждого агента 3 слоя: стратегии оптимизации, рефлексия, память
- Orchestrator — единственная шина данных; пользователь участвует
  только на 2 чекпойнтах (одобрение кредитов, одобрение питчей)
- CRM Writer — единственный, кто пишет в Excel
- Outreach Writer читает из CRM напрямую, не через JSON-контракт

Файлы для ревью:
- agents/README.md — архитектура и связи
- agents/*/AGENT.md — промты всех 6 агентов
- contracts/*.json — JSON-схемы + примеры
- config/agent-config.md — общий конфиг
- logs/*/_template.* — шаблоны памяти
</context>

<task>
Проведи ревью в 4 этапа:

**Этап 1: Целостность архитектуры**
- Прочитай README.md. Соответствует ли описанная архитектура
  реальному содержимому файлов?
- Все ли агенты из README существуют как AGENT.md?
- Все ли контракты из README существуют как JSON-файлы?
- Нет ли ссылок на несуществующие файлы?

**Этап 2: Связность потока данных**
- Пройди по цепочке: Searcher → Qualifier → Enricher → CRM Writer → Outreach Writer
- Для каждого перехода проверь: выход агента N совпадает со входом
  агента N+1? Поля в контрактах совместимы?
- Проверь примеры в контрактах: соответствуют ли они схеме?
- Orchestrator передаёт данные — есть ли в его промте достаточно
  информации, чтобы правильно парсить и маршрутизировать JSON?

**Этап 3: Качество промтов агентов**
Для каждого AGENT.md проверь:
- Цель конкретная и измеримая?
- Ссылка на SKILL.md указывает на конкретные секции (не "прочитай всё")?
- Стратегии оптимизации уникальны для этого агента (не generic)?
- Рефлексия содержит конкретные критерии оценки?
- Память описывает что читать и что писать?
- Нет ли дублирования SKILL.md (правил, таблиц, списков
  которые уже есть в скилле)?
- Нет ли "границ" (списков "НЕ делай X — это другой агент")?

**Этап 4: Consistency check**
- Пути в config/agent-config.md — все относительные?
- Язык: JSON на английском, коммуникация на русском?
- Шаблоны в logs/ — покрывают все поля, которые агенты
  обещают записывать?
</task>

<output_format>
Ответь структурированным отчётом:

## Результат ревью

### CRITICAL (блокирует запуск)
Для каждого: файл, строка/секция, что не так, как исправить.

### IMPORTANT (может сломать при определённых сценариях)
Для каждого: файл, что не так, рекомендация.

### SUGGESTIONS (улучшения, не баги)
Краткий список.

### ВЕРДИКТ
Одно из: ✅ Ready to run | ⚠️ Fix critical first | ❌ Needs rework

Если нашёл 0 critical и 0 important — напиши это явно.
Не придумывай проблемы ради объёма.
</output_format>
